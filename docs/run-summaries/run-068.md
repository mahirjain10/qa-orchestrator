## What went wrong
- `wg.Add(1)` was inside goroutine after early returns — moved before `go` keyword.
- Initial session store had data race between `UpdateStatus` (write) and `Get` (clone/read) — fixed with clone-before-mutate pattern.
- `runCampaignWithContext` signature needed `traceStore` and `artifactStore` parameters for per-flow engine creation.

## What to watch for
- `parallel_limit` defaults to 1 (sequential) — must be set > 1 in campaign config for true parallelism.
- Browser contexts share OS resources — limit concurrent contexts based on system capacity.
- Dependency context extracts URLs via regex from flow goals — only works if goals contain URLs.
- Session store clone-before-mutate adds JSON marshal/unmarshal overhead per update.
- Pause stops ALL new flow starts; running flows continue to completion.

## Key decisions
- Interface approach for ToolRegistry — preserves mock mode compatibility.
- Per-flow BrowserContext (not separate browser instances) — lighter weight, shares OS resources.
- Dependency context in system prompt (not just user prompt) — LLM sees it as a rule, not just context.
- Clone-before-mutate in session store — simpler than fine-grained field-level locking.
- Semaphore before wg.Add — ensures goroutines that early-return don't leak WaitGroup counts.
