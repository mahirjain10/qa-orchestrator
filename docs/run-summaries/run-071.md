## What happened

### Critical Bug: formatObserveUIObservation type mismatch (planner.go:117)
**Problem:** `obs.State["data"].(string)` type assertion never succeeds in production because the real `observe_ui` tool returns `map[string]any` (registry.go:253), not a JSON string. The function silently produced empty observations — the LLM had zero element data. Tests passed because the test mock returned a string, creating a test-production gap.

**Fix:** Handle `map[string]any` first (production path), then fall back to string parsing (edge case). Refactored the type assertion chain to try `map[string]any` before `string`.

### Bug 2: Test mock return type mismatch (engine_test.go:419)
**Problem:** Test mock returned `string` while production returns `map[string]any`. Tests verified behavior that doesn't exist in production.

**Fix:** Changed `observeUIToolRegistry.Execute()` to return `map[string]any` matching the real tool's return type. Updated all planner tests to use `map[string]any` in observation State. Added `TestFormatObserveUIObservation_FromStringJSON` to verify string fallback still works.

### Bug 3: autoObserve called for guided flows (engine.go:500)
**Problem:** `executeAndValidate` is called from both guided and autonomous flows. Unconditional `autoObserve` ran for every step in guided flows where observations are never read by the planner — unnecessary Playwright `evaluate()` overhead.

**Fix:** Added `ctx.Mode != sharedtypes.FlowModeAutonomous` guard at the start of `autoObserve()`. Added `TestAutoObserve_SkippedInGuidedMode` to verify.

### Bug 4: observe_ui error silently swallowed (engine.go:519-521)
**Problem:** If `observe_ui` fails (page crashed, browser disconnected), the error was silently discarded. The planner proceeded with no observation and no indication that observation failed.

**Fix:** Added `trace.EmitAgentDecision()` on error with action `auto_observe_failed` and the error message. Now visible in trace logs.

### Bug 5: Unbounded observation growth (engine.go:532)
**Problem:** Every step appended a new observation. For a 20-step autonomous flow, that's 20+ observations growing the LLM prompt context unboundedly.

**Fix:** Added `maxObservations = 10` cap. After appending, if observations exceed 10, the oldest entries are dropped. Added `TestAutoObserve_CapsObservations` to verify.
