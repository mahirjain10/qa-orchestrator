## What happened

### 1. Inconsistent Advance behavior (guided vs autonomous retry)
- **File:** `packages/agents/engine/engine.go:686,709`
- **Root cause:** In `runAutonomousFlow`, the `Retry` and `Replan` recovery handlers called `autonomousPlanner.Advance(plan)` and `stepCount++`, incrementing the plan position even though the step was being retried (not advanced). In guided mode, the equivalent paths correctly skip `Advance()`.
- **Fix:** Removed `Advance()` and `stepCount++` from both `RecoveryActionRetry` and `RecoveryActionReplan` cases in the autonomous flow.

### 2. cancelCh buffer of 1 can drop cancel signals
- **File:** `packages/runtime/lifecycle.go:84-96`
- **Root cause:** `RequestCancel` used `select { case c.cancelCh <- struct{}{}: default: }` which silently drops the signal if the channel buffer is full. The `from == to` guard in `transitionValid` allowed `Cancellingâ†’Cancelling` transitions, so a second `RequestCancel` would hit the `default` branch and drop the signal.
- **Fix:** Added early return `if c.status == RunStateCancelling { return true }` before the transition check. Changed the send to a blocking `c.cancelCh <- struct{}{}` â€” since the consumer goroutine is always running before `RequestCancel` is callable, the buffer of 1 is sufficient and double-sends are prevented by the early return.

### 3. SetStatus(Running) races with RequestCancel
- **File:** `packages/runtime/lifecycle.go:58-68` + `packages/agents/engine/engine.go:216-236`
- **Root cause:** `RunFlow` called `SetStatus(RUNNING)` and then separately checked `GetStatus() == Cancelling`. Between those two calls, `RequestCancel` could fire, making the check miss the cancellation.
- **Fix:** Added `LifecycleController.BeginExecution() (<-chan struct{}, bool)` that atomically transitions to RUNNING and returns the cancel channel. The engine uses a non-blocking `select` on the returned channel to catch any cancel signal that arrived during (or immediately after) the state transition.

### 4. FindStepByID returns pointer into slice array
- **File:** `packages/agents/engine/engine.go:975-982`
- **Root cause:** `findStepByID` returned `&steps[i]` â€” a pointer into the slice's backing array. If the slice were later appended to, the backing array could be reallocated, making the pointer stale.
- **Fix:** Changed signature to `findStepByID(steps, stepID) (Step, bool)` â€” returns a value copy + found flag. Updated all callers and tests.

### 5. DependencyContext passed to BuildUserPrompt but unused
- **File:** `packages/llm/prompts.go:157-165` + `packages/agents/planner/planner.go:238`
- **Root cause:** `PlannerPromptData.DependencyContext` was set in the caller but never read by `BuildUserPrompt`. The field was only used in `BuildSystemPrompt` (which receives it as a separate parameter).
- **Fix:** Removed `DependencyContext` field from `PlannerPromptData` struct and its sole assignment in planner.go.

### 6. Inconsistent sanitizeID across stores
- **File:** `packages/storage/trace/store.go:145-150`
- **Root cause:** Session and artifact stores replaced `/`, `\`, `..`, and `:` with `_`. The trace store was missing the `":" â†’ "_"` replacement, meaning run/flow IDs containing colons could produce Windows-illegal filenames.
- **Fix:** Added `clean = strings.ReplaceAll(clean, ":", "_")` to trace store's `sanitizeID`.

### 7. Log file handle never closed
- **File:** `apps/tui/cmd/main.go:46-47`
- **Root cause:** `logging.InitFileOnly()` opens `./logs/app.log` but `logging.Close()` was never called.
- **Fix:** Added `defer logging.Close()` immediately after the `InitFileOnly` call.

### 8. UI double cursor in command bar
- **File:** `apps/tui/internal/components/command_bar.go:119`
- **Root cause:** `m.Input.View()` renders a textinput with its own built-in cursor. Appending `"â–ˆ"` after it produced two adjacent cursors.
- **Fix:** Removed `+ "â–ˆ"` from the `Render` call.

### 9. runCreatedCmd only fires once
- **File:** `apps/tui/internal/screens/main.go:188-199`
- **Root cause:** `runCreatedCmd` is a one-shot Bubble Tea command that reads from the channel once. After the first `runCreatedMsg`, the command is never re-registered, so subsequent campaign starts via the same channel are ignored.
- **Fix:** In the `runCreatedMsg` handler, re-register `runCreatedCmd(m.runCreatedCh)` so the channel watcher persists across multiple campaign creations.
