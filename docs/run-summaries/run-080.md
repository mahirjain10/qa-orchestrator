## Goal
Fix 3 bugs observed in Run 079: (1) LLM hallucinates selectors like `a:has-text("Practice")` that don't match observed elements. (2) LLM API returns empty responses exhausting all 3 retries. (3) 404 detection title check flags pages where "404" appears anywhere in the title.

## What happened

### Fix A: Auto-replace hallucinated selectors (engine.go)

**Problem:** Safety Net 4 (selector validation, engine.go:442-464) caught hallucinated selectors and injected a steering instruction, but the LLM often ignored it or kept hallucinating. The root cause: the LLM is good at identifying *what* to click but bad at formatting exact CSS selectors.

**Fix:** When Safety Net 4 detects an invalid selector, it now:
1. Extracts the intended text from `tag:has-text("Intent")` pattern using regex
2. Searches observed elements for one whose `text` field contains the intent (case-insensitive substring)
3. Among matches, prefers the element with the closest text-length ratio (intent / element_text)
4. If found → auto-replaces the selector in the planStep and proceeds to execution
5. If not found → falls through to existing steering instruction approach

**New functions added:**
- `extractTextFromSelector(selector string) string` — parses `:has-text("Foo")` → `"Foo"`
- `observedElements(observations []Observation) []map[string]any` — returns full element data from most recent observe_ui
- `findBestMatchSelector(intentText string, elements []map[string]any) (string, bool)` — finds best match by text content ratio, preferring anchor tags and shorter text as tiebreakers

### Fix B: Model fallback for empty LLM responses (config.go + client.go)

**Problem:** The LLM API consistently returned empty responses for gpt-5-mini on OpenRouter. All 3 retry attempts failed with the same empty response. Retrying the same failing model is futile.

**Fix:** Implemented model fallback chain:
1. Added `FallbackModels []string` to `Config` struct
2. Added `LLM_FALLBACK_MODELS` env var (default: `openai/gpt-4o-mini,gemini/gemini-2.0-flash-001`)
3. Modified `Generate` to loop over `[primaryModel] + fallbackModels`
4. Only triggers fallback for empty responses (`errors.Is(lastErr, ErrEmptyResponse)`) — other errors (auth, timeout) bail immediately
5. Added `providerForModel(model)` helper that detects the correct provider and applies config (API keys, headers, referers)
6. Added `ErrEmptyResponse` sentinel for reliable error matching

### Fix C: Tighten 404 detection (registry.go)

**Problem:** The document.title check (`strings.Contains(titleLower, "404")`) fired first and flagged legitimate pages whose title mentioned "404" in passing (e.g., "Learn about 404 errors — Tutorial").

**Fix:**
1. Swapped check order — heading check runs FIRST (more targeted)
2. Changed title check from `strings.Contains` to `strings.HasPrefix` — only flags if title starts with "404" or "not found"
3. Heading check remains `strings.Contains` (headings are naturally short and specific)

## Key decisions
- **Fix A auto-replace is the strongest fix** because it bypasses the LLM entirely for selector syntax. The LLM decides *what* to click via intent text, but the code handles *how* to format the selector. This is a 2-layer defense: (1) text extraction from selector pattern, (2) element text matching from observation.
- **Model fallback only triggers for empty responses**, not other errors. If a model returns auth errors or timeouts, other models on the same provider would likely fail the same way. Empty responses are usually model-overload or provider-specific saturation.
- **Default fallback models prioritize OpenAI then Gemini** — they have independent failure modes (OpenRouter might be saturated but Gemini direct API might work).
- **Fix C uses `HasPrefix` not `Contains`** for the title check — legitimate 404 pages typically start with "404" or "Not Found" in the title. A title like "Handling 404 Errors — Tutorial" is not a real 404 page.
