## Goal
Fix two bugs observed in real browser runs of `large-parallel.yaml`: (1) the Yo-Yo loop where the LLM re-navigates to a URL that just produced a 404, and (2) OpenAI empty response errors crashing flows without retry. Also fix 404 false positive detection.

## What happened

### Fix 1: Yo-Yo Loop (Prompt Engineering)
Added explicit "NEVER use navigate() again with a URL that just produced a 404" rule to both the system prompt (`prompts.go:64-71`) and user prompt (`prompts.go:100-101`). After observing the root domain, the LLM is now forced to use `click()` on visible links instead of re-navigating to the dead URL.

### Fix 2: Empty LLM Response Retry
Changed `packages/llm/client.go:178` from `NewNonRetryableError` to `NewRetryableError(..., true)`. Empty responses from OpenAI/OpenRouter are now retried with exponential backoff (500ms → 1s → 2s) up to 3 times before failing the flow.

### Fix 3: 404 False Positives
Replaced the broad `body.innerText.substring(0,200)` substring check in `registry.go:337-349` with a targeted `document.querySelector('h1, h2, .error, .not-found')` element query. This prevents real pages (students, file-upload, timeouts) from being falsely flagged as 404 just because they contain the text "404" or "not found" in navigation links or sidebars.

## What to watch for
- The prompt fixes may need tuning if the LLM still finds ways to re-navigate to 404 URLs (e.g., using a slightly different URL). Monitor the next run logs.
- The retry adds latency (up to ~3.5s total for 3 retries with exponential backoff). If transient empty responses are frequent, consider reducing retries to 2.
- The 404 detection change may miss some legitimate 404 pages if their headings don't use standard elements (`h1`, `h2`, `.error`, etc.). Monitor for false negatives.

## Key decisions
- Chose prompt engineering for the Yo-Yo fix over code-level enforcement because the LLM needs flexibility in how it navigates the site. A code-level ban on `navigate()` would be too restrictive and prevent legitimate re-navigation to different URLs.
- Made empty responses retryable (not non-retryable) because the traces showed these are transient API glitches, not content filtering. All 3 retry attempts in the test produced the same empty response, confirming the mock was correct.
- Chose heading/element queries over URL-based heuristics for the 404 detection because URL-based approaches would require knowledge of which URLs are "real" at each site.
