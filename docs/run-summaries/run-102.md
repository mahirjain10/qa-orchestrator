## What happened

### S1: Inconsistent sanitizeID across stores (`trace/store.go:145-151`)
- **Bug:** session/artifact used `strings.NewReplacer`, trace used sequential `strings.ReplaceAll`. Same result but different implementation — inconsistent.
- **Fix:** Rewrote trace's `sanitizeID` to use `strings.NewReplacer` matching session/artifact pattern.

### S2: ListRunIDs reads filesystem without lock (`trace/store.go:106`)
- **Bug:** `ListRunIDs()` read the filesystem directory without acquiring `s.mu`, risking race with concurrent writes.
- **Fix:** Added `s.mu.RLock() / defer s.mu.RUnlock()` at the top of `ListRunIDs()`.

### S3: Save writes file before persisting index (`artifact/store.go:74-93`)
- **Bug:** `os.WriteFile()` at line 74, then in-memory state updated at lines 91-92, then `persistIndex()` at line 93. Crash between WriteFile and persistIndex orphaned the file.
- **Fix:** If `persistIndex()` fails, roll back in-memory state (remove from `artifacts` slice and `index` map) and delete the orphaned file. Caller can retry cleanly.

### S4: overflowDropped dead field (`lifecycle.go:20`)
- **Bug:** `overflowDropped int` field declared on `LifecycleController` struct, never read or written.
- **Fix:** Removed the field entirely.

### S5: runWithContext goroutine leak (`runtime.go:750-767`)
- **Bug:** On context cancellation, spawned a drain goroutine `go func() { <-ch }()` to unblock fn's channel send. With a buffered channel (cap 1), this is unnecessary — fn's send always succeeds.
- **Fix:** Removed the drain goroutine. On `ctx.Done()`, just `return ctx.Err()`.

### S6: BrowserRuntime.ensurePage doesn't check isRunning (`runtime.go:218`)
- **Bug:** After `Stop()`, stale page/context references could succeed in `ensurePage()`.
- **Fix:** Added `if !r.isRunning { return fmt.Errorf("browser not running") }` at the top of `ensurePage()`.

### S7: FlowBrowserRuntime.ensurePage has no recovery (`runtime.go:538`)
- **Bug:** Unlike parent `BrowserRuntime.ensurePage()` which recreates page/context on close, `FlowBrowserRuntime.ensurePage()` returned error immediately.
- **Fix:** Added 2-tier recovery matching parent pattern: (1) try `context.NewPage()` on existing context, (2) try `parent.browser.NewContext()` + `NewPage()`.

## What to watch for
- S7 FlowBrowserRuntime recovery accesses `r.parent.browser` without parent lock — same pattern as existing `getTimeout()`, but could race with parent `Stop()`. A broader parent-lock refactor would be needed to fix this systematically.
- S3 rollback uses `os.Remove` inside the mutex — fine for correctness, but the removed file will be lost. The caller sees an error and can retry.

## Key decisions
- S5: Removed drain goroutine entirely (buffered channel makes it safe). A blocked `fn` goroutine can still leak, but that's `fn`'s responsibility.
- S7: Recovery mirrors parent `BrowserRuntime.ensurePage()` pattern: try page recreation first, then context recreation.
- S3: Rollback approach (delete file + revert in-memory) chosen over order reversal because `persistIndex()` reads in-memory state, which must be updated first.
