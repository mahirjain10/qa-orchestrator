## What went wrong
- Previous implementation of `createLLMClient` manually parsed environment variables, completely bypassing `llm.Config.Validate()`. If keys were missing, it silently returned `nil` instead of bubbling up the error.
- As a result, the TUI would start up normally, the `AgentEngine` would hit a `nil` LLM client, immediately mark the flow as `FAILED`, and the user wouldn't clearly understand why the engine "got stuck" or immediately errored out.

## What to watch for
- `llm.LoadConfig()` is now the unified source of truth for loading the API Key, Model, and configuring retries/timeouts.

## Key decisions
- Panicking early is the desired behavior for a CLI tool if required configurations are missing before it attempts to do real work.
