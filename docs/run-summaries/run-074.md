# Run Summary — 074

## Run
- ID: 074
- Date: 2026-05-22
- Agent: opencode

## Goal
Fix two bugs from the real browser run (run_20260522_6190d7): (1) `max_steps_reached` reporting PASSED instead of FAILED when autonomous steps exhausted, (2) repeat detection having no hard-break — LLM could repeat the same step indefinitely until maxSteps consumed.

## What happened
Analyzed real LLM trace logs and identified two systematic bugs and one additional false-positive issue:

**Bug 1 — max_steps_reached → OutcomePass:** The `if stepCount >= maxSteps` block at line 561 emitted a trace event but never set `result.Outcome` or appended to `result.Errors`. Since `result.Outcome` defaults to `OutcomePass` and the `done:` safety check (`OutcomePass && len(Errors) > 0 → OutcomeFail`) evaluates to false with empty errors, the flow was always marked as COMPLETED/PASSED even after burning all 20 steps on useless repeats.

**Bug 2 — repeat detection never hard-breaks:** The repeat-detection safety-net at line 450-458 injected a steering instruction and `continue`d — always. With no counter or threshold, the LLM could regenerate the same tool+params every time, steering instructions rotated out after 5, and stepCount burned up until maxSteps was reached. Flow-zeta called `get_text(#password)` 18 times before exhausting the step budget.

**Bug 3 — safety-net 4 false positives on structural selectors:** after fixing bugs 1 and 2 and running `make run-real-large`, 3 more flows failed because safety-net 4 blocked `get_html(selector:table)`. The `observe_ui` tool only returns interactive elements (links, inputs, buttons). Structural selectors like `table`, `form`, `div` are never observed, yet they're perfectly valid for read-only DOM queries. The block should only apply to interaction tools (click, type_text, wait_for) that can timeout on phantom selectors.

### Fixes
- **Bug 1:** `consecutiveRepeats` counter declared and initialized to 0 alongside `stepCount`
- **Bug 1:** Hard-break block: `if consecutiveRepeats >= 3 { OutcomeFail; goto done }`
- **Bug 2:** `consecutiveRepeats = 0` on the non-repeat path where `LastStepSignature` is set
- **Bug 2:** `max_steps_reached` block: now sets `OutcomeFail` + `"max autonomous steps reached without finishing"`
- **Bug 3:** safety-net 4 now uses a `switch` on `planStep.Tool` — read-only tools (`get_html`, `get_text`, `inner_html`, `evaluate`) skip selector validation entirely. Interaction tools (`click`, `type_text`, `hover`, etc.) still get validated.

## What to watch for
- The hard-break uses `goto done`, which skips the `stepCount >= maxSteps` block — intentional, avoids double-triggering when both conditions are true
- Safety-net thresholds (repeat count = 3) may need tuning after more real runs
- Steering instruction cap (5) is unchanged — if LLM ignores 3 steering messages, the hard-break fires regardless

## Key decisions
- Hard-break uses `goto done` (not `break`) to skip the `stepCount >= maxSteps` check and avoid double-triggering
- Counter resets on ANY non-repeat step, even if that step fails — prevents the LLM from being punished for past repeats after changing approach
- Error strings are distinct and greppable so trace log analysis immediately identifies the failure mode
