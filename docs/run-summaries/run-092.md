## Goal
Fix 5 critical campaign reliability failures (API timeouts, broken retry logic, uncontrolled step repetition, JSON parsing failures, state staleness) and 2 edge cases regarding the LLM "thinking" budget configuration.

## What happened
1. **API Timeouts (P0 Reliability):** Increased `defaultTimeout` from `30` to `120` seconds in `packages/llm/config.go`. Reasoning models like DeepSeek-V4-Pro routinely exceed 30 seconds during their thinking phase, leading to premature context cancellation.
2. **Broken Retry Logic (P0 Reliability):** Replaced `flowEngine.RunFlow` with `flowEngine.RunFlowWithRetry` in `apps/tui/cmd/main.go`, ensuring the engine correctly respects `f.Config.RetryLimit` (or falls back to the campaign config limit). 
3. **Uncontrolled Step Repetition (P1 Reliability):** Reduced the `consecutiveRepeats` hard-break threshold from `3` to `2` in `packages/agents/engine/engine.go`. Wasting 3 complete cycles on an identical failing step was too lenient.
4. **JSON Parsing Failures (P1 Reliability):** In `runAutonomousFlow` (`packages/agents/engine/engine.go`), intercepted errors containing `"parsing"`. Instead of instantly failing the flow, the engine now injects a steering instruction (`"Your previous response was not valid JSON. Please return ONLY a valid JSON array..."`) and continues the loop to allow the LLM to self-correct.
5. **State Staleness (P2 Reliability):** Added an explicit `sessionStore.UpdateFlowState(runID, fid, sharedtypes.FlowStatePassed, "")` in the flow worker goroutine inside `apps/tui/cmd/main.go` when `result.Outcome == engine.OutcomePass` to ensure the final success state is atomically persisted.
6. **Gemini Thinking Budget Edge Case:** Fixed `provider_gemini.go` to gate the `geminiThinkingConfig` on `req.Thinking.Type == "enabled"` rather than requiring `BudgetTokens > 0`. This allows users to enable thinking while trusting the model's default budget.
7. **MaxTokens Cap Edge Case:** Fixed `client.go` to only default `MaxTokens` to 1024 if `req.Thinking` is explicitly nil, disabled, or empty. This prevents the default token cap from truncating reasoning models during their thinking phase.

## What went wrong
- Previous runs implemented retry config but failed to wire `RunFlowWithRetry` into the orchestrator.
- The default 30-second timeout was a legacy value from fast, non-reasoning models and was fundamentally incompatible with DeepSeek/o1 models.

## What to watch for
- Reasoning models take significantly longer to respond (up to 60-90s). The `defaultTimeout` of 120s is necessary, but the UI may appear "stuck" while waiting.
- The JSON parsing steering instruction is appended to the prompt; if the LLM continues to fail JSON parsing, it will hit the max autonomous steps limit.

## Key decisions
- Opted to inject a steering instruction for parsing failures rather than immediately failing the flow, leveraging the LLM's ability to correct syntax errors when prompted.
- Reduced the repeat limit to 2 to save tokens and time; if the LLM repeats a failure twice, it is unlikely to succeed on the third identical attempt.
