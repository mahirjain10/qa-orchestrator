## Goal
Fix Fix B — the model fallback implementation from Run 080 was switching providers for each fallback model, routing requests to the native provider's endpoint instead of the user's configured endpoint (OpenRouter). This caused 401 "Missing Authentication header" errors because OpenRouter API keys don't work on `api.openai.com`.

## What happened

**Diagnosis from real browser traces:**
- Pre-Fix 080 run (_9e360c): expected behavior — flows targeting real pages (login, exceptions) pass; flows targeting non-existent pages (students, file-upload, timeouts) correctly fail with 404 → root domain recovery
- Post-Fix 080 run (TUI output): 5/6 flows failed with 400/401 errors — **worse than before**
- Root cause: `providerForModel("gpt-4o-mini")` returned `OpenAIProvider{Endpoint: "https://api.openai.com/v1/chat/completions"}` but the user's API key is for OpenRouter. Fallback requests went to OpenAI's real API → 401.

**Fix:**
- Removed `providerForModel()` entirely — all models use the same provider
- Removed provider-switching block from `Generate()` fallback loop
- OpenRouter can handle any model name; the only thing that changes is `req.Model` in the request body
- Removed 2 tests: `TestProviderForModel` and `TestGenerateModelFallback_ProviderDetection` (tested removed functionality)
- All 37 remaining tests pass

## What went wrong
- **Run 080 Fix B was fundamentally broken**: `providerForModel` created a *new provider* for each fallback model, routing to that provider's hardcoded endpoint. Users whose primary model goes through OpenRouter would have fallback requests sent to OpenAI's native API where their OpenRouter key doesn't work.
- The model-detection approach (`detectProviderName` → `GetProvider`) is correct for *initial setup* but wrong for *fallback logic* because the endpoint/auth should stay the same.

## What to watch for
- `detectProviderName` still exists and is used for initial provider setup (config.go, provider.go). That's correct — initial setup needs to pick the right provider for the primary model.
- Don't re-add dynamic provider switching in the fallback path. If a user needs different providers for different models, they should route through OpenRouter (which is model-agnostic).
- The existing `ErrEmptyResponse` + `NewRetryableError` fallback trigger is correct and unchanged.

## Key decisions
- **Same provider for all models** — simpler, matches how OpenRouter/aggregator APIs work. The endpoint and auth headers don't change, only the model name.
- **Removed `providerForModel` entirely** — it was only used in the broken fallback path. Keeping dead code invites confusion.
- **Kept `detectProviderName`** — still used for initial provider selection during startup.
