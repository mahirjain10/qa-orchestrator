## What went wrong
- Go workspace (go.work) with nested modules caused complex `replace` directives.
- Race condition in session store: `loadFromFile` was modifying map inside read lock.
- Multiple goroutines accessing session store could cause panic.

## What to watch for
- Never write to shared state while holding a read lock (RLock).
- Separate read-only operations (loadFromFile) from write operations.
- Use double-checked locking pattern: acquire RLock first, then WLock if needed.
- Go workspace files (go.work) should be avoided for simple projects.

## Key decisions
- Single root module instead of workspace to eliminate the need for `replace` directives.
- `loadFromFile` made purely functional (no side effects).
- All cache updates happen under Write Lock (Lock) only.
