## Goal
Fix 3 concurrency bugs identified in the codebase: (1) Context cancellation completely ignored in browser runtime Playwright calls. (2) Semaphore blocking in flow goroutine defeats pause/cancellation. (3) TOCTOU race condition in LifecycleController.RequestCancel.

## What happened

### Bug 1: Context Cancellation Ignored in Browser Runtime
- Added `ctx context.Context` to all `BrowserRuntimeInterface` methods: `Click`, `Fill`, `WaitForSelector`, `TextContent`, `InnerHTML`, `Evaluate`, `Screenshot`.
- Only `Navigate` already had context â€” all other Playwright driver calls had no way to check cancellation.
- Each method now checks `ctx.Err()` immediately before the Playwright driver call returns, preventing hangs when the user presses 'x' to cancel.
- Updated both `BrowserRuntime` and `FlowBrowserRuntime` implementations.
- Updated `ToolRegistry` registration closures to pass `r.ctx` to all runtime calls (17 call sites).
- Updated `mockRuntime` in test file to match the new interface.

### Bug 2: Semaphore Blocking Defeats Pause & Cancellation
- Replaced `semaphore <- struct{}{}` with `select { case semaphore <- struct{}{}: case <-ctx.Done(): ... }` in the flow goroutine.
- When the context is cancelled while a flow goroutine is waiting for the semaphore slot, it now exits immediately instead of blocking until acquiring the slot.
- The cancellation handling block updates session status and flow state to cancelled, matching the existing pattern used earlier in the same function.

### Bug 3: Race Condition in LifecycleController.RequestCancel
- `CanCancel()` was evaluated outside the mutex lock, opening a TOCTOU window where the status could transition to `Completed` between the check and the write.
- Fixed by combining the state check and write under a single `c.mu.Lock()`.
- Removed the `CanCancel()` call from `RequestCancel()` â€” the switch statement handles it directly.
