## What went wrong

## What to watch for
- 404 check is now FIRST in the error chain ‚Äî any error on a 404 page triggers retry+steering, regardless of error type. This is intentional: a 404 page invalidates all element interactions.
- The `has404Warning` function checks `State["data"]`, `State["warning"]`, and `State["last_output"]` ‚Äî covers all observation storage paths.
- Steering instructions are capped at 5; the 404 steering will evict older instructions if needed.
- The `observe_ui` 404 detection (registry.go lines 337-349) uses page title + body text substring matching ‚Äî fragile for non-English sites or custom 404 pages styled differently.

## Key decisions
- **404 check first**: Placed before all other error checks so any failure on a 404 page is attributed to the missing page rather than the specific element. If the LLM navigates to root domain and fails there, normal error handling applies.
- **Three-path observation scan**: `has404Warning` checks `State["data"]` (the path used by `injectObserveStep` and `autoObserve`), `State["warning"]` (direct warning), and `State["last_output"]` (validator's `CreateObservation` path). This covers all possible storage locations.
- **Did NOT modify engine.go**: The steering instruction injected by `Decide()` into `ctx.SteeringInstructions` is automatically picked up by `drainSteeringEvents` before the next LLM call. No engine changes needed.
