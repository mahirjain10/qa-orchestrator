## Goal
Fix 3 security/correctness bugs: (4) File path injection in storage stores. (5) Upstream OutcomeFail ignored in dependency evaluation. (6) LLM fallback models logically defeated by restrictive guard.

## What happened

### Bug 4: File Path Injection / Arbitrary Directory Traversal
- **Root cause:** `session/store.go`, `artifact/store.go`, and `trace/store.go` blindly concatenated `runID` and `flowID` into file paths. A malicious ID like `../../../etc/passwd` would escape the data directory.
- **Fix:** Added `sanitizeID()` function to all three stores that strips `..`, `/`, and `\` (replaced with `_`).
- Applied in `session/store.go:filePath()`, `artifact/store.go:artifactDir()`, and `trace/store.go:filePath()`.
- Also added `"strings"` import to artifact store which was missing it.

### Bug 5: Dependency Evaluation Failure (Upstream Failures Ignored)
- **Root cause:** Line 425 in `main.go` only checked `depResult.Outcome == engine.OutcomeSkip` ‚Äî flows whose dependencies failed with `OutcomeFail` were treated as unblocked and executed.
- **Fix:** Added `|| depResult.Outcome == engine.OutcomeFail` to the upstream failure check.
- DAG constraints are now fully enforced for both `Skip` and `Fail` outcomes.

### Bug 6: LLM Fallback Models Are Logically Defeated
- **Root cause:** Line 107-112 in `client.go` had `if modelIdx < len(models)-1 && !errors.Is(lastErr, ErrEmptyResponse) { return ... }` ‚Äî this aborted fallback for any error that wasn't an empty response. HTTP 502, 503, and 429 errors would immediately fail instead of trying fallback models.
- **Fix:** Removed the restrictive guard. Non-retryable errors (auth, validation) already return early before this point, so any error reaching here is retryable and should trigger fallback.
- Removed unused `"errors"` import.

## What to watch for
- `sanitizeID` replaces path separators with `_` ‚Äî this changes the on-disk filename. If any external tool reads these files by constructing paths, it must also sanitize.
- The LLM fallback change means the system will try fallback models more aggressively ‚Äî ensure `LLM_FALLBACK_MODELS` contains working model names for your provider.
- The `TestEmptyResponseIsRetryable` test in `client_test.go` should still pass since empty responses remain retryable.

## Key decisions
- `sanitizeID` uses simple string replacement rather than `filepath.Clean()` ‚Äî replacement is more predictable and doesn't rely on OS path semantics.
- Same `sanitizeID` function duplicated in each store package rather than shared ‚Äî avoids import coupling and keeps changes self-contained.
- For Bug 6, chose `continue` over removing the guard entirely ‚Äî the outer loop naturally iterates to the next model, and the existing logging at line 76-81 provides visibility.
