## Goal
Fix TUI visual rendering bugs (`%!d(string=RUNNING)s`), auto-refresh locking, and verify fail-fast behavior.

## What happened
1. Fixed the Flow Status rendering bug: Rewrote the string formatting logic in `apps/tui/internal/components/flow_status.go`. The previous implementation was passing lipgloss ANSI strings into a nested `fmt.Sprintf()` which corrupted Go's internal string parsing, leading to `%!d(...)` formatting panics in the UI.
2. Fixed the Auto-Refresh (Live Polling) bug: Found that Bubbletea's `tea.Tick` is a one-shot command. When `TickMsg` was received in `main.go`, the previous implementation returned `nil` instead of queueing the *next* `tea.Tick()`. This caused the UI to completely freeze and never update after the first frame (unless the user manually pressed `r`). Re-queued the tick inside the update loop.
3. Verified the fail-fast logic: It works perfectly. The reason it wasn't triggering for the user is that they launched the TUI without a new campaign, so the parser never ran the LLM check. Instead, the TUI loaded *old* history from the `data/` directory that were visually "stuck" in a RUNNING state due to the auto-refresh bug mentioned above.

## What went wrong
- Passing Lipgloss strings containing ANSI escape codes into an outer `fmt.Sprintf` layout string is a known trap in Go string formatting.
- Forgetting to recursively return `tea.Tick` in a Bubbletea Update loop will silently kill background polling.

## What to watch for
- Always ensure format strings (`%s`, `%-10s`) are resolved *before* wrapping them in Lipgloss coloring to avoid ANSI code corruption.

## Key decisions
- Extracted nested formatting into separate clean `fmt.Sprintf` operations to guarantee safe rendering.
